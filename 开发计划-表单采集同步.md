# 经费年报生成工具 — 关键数在线采集与同步开发方案

## 背景与目标

民办学校 / 无报表学校没有五件套源文件，只能靠"关键数 + 上年结构拆分"生成草稿
（现有 `computePrivateDraft`）。目前关键数要在桌面端逐校手工录入，效率低、易错。

目标：

1. 学校（主要是民办园、无报表学校）通过手机/电脑在线填写关键数；
2. 桌面软件一键同步采集结果；
3. 合并填报的学校群（中心园 + 成员园）自动汇总成员数据，按中心园出一份草稿；
4. 同步后批量生成草稿，全程留痕可追溯。

---

## 总体架构

```
学校（专属链接填表，手机可填）
        │ 提交
        ▼
自建服务器（表单页 + SQLite + 拉取接口，挂 jyj.yunbg.vip 子路径或新子域）
        │ 桌面端 token 拉取
        ▼
桌面软件（同步 → 匹配上年年报/教育事业年报 → 合并组汇总 → 批量生成草稿）
        ▼
草稿年报 xlsx（预览复核后上报平台）
```

## 关键设计决策（先定调，避免返工）

### D1. 桌面端是名单和规则的唯一来源

学校名单、合并关系（`regionRules.mergeGroups`）、已撤销学校（`ignoredClosedSchools`）
全部以桌面端配置为准，由桌面端**推送**到服务器。服务器只是哑采集器，
不维护第二份合并规则，避免两处规则漂移。

名单生成入口：现有 `get-edu-merge-summary` 已能输出「合并中心园 + 成员园 +
独立民办园」，直接用它生成采集名单。

### D2. 每校一个专属带码链接，学校名不可改

`/f/:code` 打开即本校表单，名称锁定。杜绝手输校名与教育事业年报对不上的问题。
链接由桌面端推送名单后批量生成，可导出为 Excel（校名 + 链接 + 二维码）供微信/QQ 群发。

### D3. 合并组：成员园各填各的，软件端汇总

- 每所成员园（含中心园自身）填**自己账上的数**——账在各园，让中心园代汇总
  既容易错又无法追溯。
- 桌面端同步后按合并组把成员数值**求和**，得到中心园的 controls，
  再用中心园的上年经费年报（本身就是合并口径）做结构拆分。
- 兜底：管理页允许中心园负责人代填某个成员园（打开该成员的链接填），
  应对个别园没人会操作的情况。

### D4. 同步方向：桌面端拉取（pull）

不做服务器推送。桌面端点"同步"或定时轮询拉取接口，实现简单、
不要求桌面端可被公网访问。

## 表单字段（与 `computePrivateDraft` controls 一一对应）

| 表单项 | controls 字段 | 必填 | 校验 |
|---|---|---|---|
| 学费/保育教育费收入 | tuitionIncome | ✅ | ≥0 |
| 财政补助收入 | fiscalSubsidy | ✅ | ≥0 |
| 全年工资福利总额 | wageTotal | ✅ | ≥0；÷教职工数落在当地区间外时前端提示 |
| 资本性支出 | capitalExpense | ✅ | ≥0 |
| 有无房租 → 金额 | hasRent / rentExpense | 开关 | 开则金额必填 |
| 有无贷款利息 → 金额 | hasLoan / interestExpense | 开关 | 同上 |
| 举办者投入 → 金额 | hasSponsorInput / sponsorInput | 开关 | 同上 |
| 举办者抽回 → 金额 | hasSponsorWithdraw / sponsorWithdraw | 开关 | 同上 |
| 捐赠收入/支出 | hasDonation / donationIncome / donationExpense | 开关 | 同上 |
| 其他收入 | otherIncome | 可选 | ≥0 |
| 是否供暖 | hasHeating | 开关 | — |
| 填表人姓名、电话 | （留痕用） | ✅ | 手机号格式 |
| 备注 | （留痕用） | 可选 | — |

前端即时校验 + 提交前汇总确认页（"贵园收入合计 X 元、支出关键数 Y 元，确认提交？"），
把返工消灭在填表现场。提交成功页不回显明细金额。

## 服务器端（新建，Node.js + Express + better-sqlite3）

与桌面端同技术栈，校验逻辑可直接复用一份 JS。

### 数据模型

```sql
schools(
  id, year, unit_name UNIQUE, fill_code UNIQUE,   -- fill_code = 随机 16 位
  merge_center,          -- 所属合并中心园名，独立园为 NULL
  contact, created_at
);
submissions(
  id, school_id, year,
  payload_json,          -- controls 原始 JSON
  filler_name, filler_phone, note,
  version,               -- 同校重复提交递增
  created_at
);
```

### 页面与接口

| 路由 | 说明 |
|---|---|
| `GET /f/:code` | 学校填表页（移动端优先，校名锁定） |
| `POST /f/:code` | 提交；服务端二次校验后入库，version+1 |
| `GET /admin` | 进度看板：按合并组分组显示已填/未填/联系人；管理 token 登录 |
| `POST /api/v1/schools/sync` | 桌面端推送名单+合并关系，返回各校 fill_code 与链接 |
| `GET /api/v1/submissions?year=&since=` | 桌面端拉取（增量），固定 token 鉴权 |

## 安全

- 拉取/推送接口固定 token（配置在桌面端，不硬编码进前端页面）
- fill_code 16 位随机、不可枚举；填表页不显示其他学校任何信息
- 全站 HTTPS（现有证书）；服务器 SQLite 每日备份
- 提交成功页不回显金额

## 风险与待确认

1. **合并组内个别园长期不填**：靠"本次忽略"兜底，但金额会缺口——
   需和业务确认：缺报成员按 0 处理还是必须催齐？
2. **中心园上年年报口径变化**（今年并入/退出了成员园）：结构拆分仍用中心园
   上年表，金额汇总用今年名单，两者口径不一致时只影响拆分比例，可接受；
   极端情况允许手工调整后用「已修正」保存。
3. **学校填报期集中在截止前两天**：服务器压力可忽略（几百所校、纯表单），
   重点是进度看板 + 未填名单催报要好用。
4. 新开办园无上年年报：本方案不解决，走「同类园结构模板」（另列计划）。
